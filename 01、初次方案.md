
## 技术特点

### 1. 三阶段碰撞检测状态机

系统摒弃传统单阈值触发方式，采用**“失重-冲击-伤后静止”三阶段状态判定模型**。状态机以 **10Hz频率** 读取LIS2DH12加速度计三轴数据，实时计算合力幅值 \(\sqrt{x^2+y^2+z^2}\)，通过纯时序比较驱动状态流转：

- **IDLE（空闲）**：常态监测，持续计算合力幅值；
- **WEIGHTLESS（失重）**：检测到合力 ≈ 0.5g 并持续 **100ms**，判定骑行姿态异常（如跌落或腾空）；
- **IMPACT（冲击）**：合力瞬时峰值 > 3.5g，判定为有效碰撞事件；
- **POST_ACCIDENT（事故后静止）**：进入 **30秒** 倒计时监测窗口，等待用户取消；
- **SOS_TRIGGERED**：30秒内无人干预，自动触发告警发送。

任意阶段超时或用户按键中断，状态机立即复位至IDLE。该设计**不依赖任何外部中断**，纯由主循环时序驱动，逻辑清晰且易于调试复现。

### 2. 30秒“二次确认”人机交互机制

碰撞触发进入POST_ACCIDENT状态后，系统启动30秒倒计时窗口，为用户提供**误报取消通道**：

- **声光反馈**：板载蜂鸣器以 **0.5Hz** 频率间歇鸣叫，LCD屏幕实时刷新剩余秒数，提示用户当前处于告警预备状态；
- **按键取消**：用户若意识清醒，按下板载按键触发GPIO外部中断（已做 **200ms** 硬件防抖处理），状态机立即复位至IDLE，终止倒计时；
- **自动告警**：倒计时归零且无人干预，系统自动跃迁至SOS_TRIGGERED状态，执行远程求助发送流程。

该机制有效平衡了**碰撞检测的灵敏度**与**日常使用的可接受性**，避免因颠簸路面误触发导致无效救援。

### 3. 基于 _thread 的异步并发软件框架

软件层基于 MicroPython 的 `_thread` 多线程模块，构建**三线程并发任务架构**，确保传感器采集、碰撞判定与网络通信互不阻塞：

| 线程 | 职责 | 运行周期 |
| :--- | :--- | :--- |
| 传感器线程 | 以10Hz频率采集LIS2DH12加速度、AHT20温湿度及光敏ADC数据，写入全局共享缓冲区 | 100ms |
| 碰撞检测线程 | 读取缓冲区加速度数据，驱动状态机判定，管理30秒倒计时逻辑 | 50ms轮询 |
| 通信线程 | 维持MQTT心跳保活，SOS触发时立即组装并发送含GPS坐标的JSON报文 | 持续运行 |

线程间通过**全局变量 + 互斥锁（`_thread.allocate_lock()`）** 实现数据同步，避免资源竞争导致的数据错乱或死锁。

### 4. GNSS + MQTT 联动上报机制

SOS触发时，通信线程执行以下联动流程：

1. **定位读取**：获取最近一次有效的GNSS定位结果，包含纬度、经度及UTC时间戳；
2. **报文组装**：按预设格式生成JSON结构化数据：
   ```json
   {
     "type": "SOS",
     "lat": 31.2304,
     "lng": 121.4737,
     "timestamp": "2025-07-02T14:30:00Z",
     "status": "collision_detected"
   }
   ```
3. **双通道上报**：通过MQTT协议发布至云端 `SOS_TOPIC` 主题，同时通过HTTP POST写入备用数据库，确保消息冗余可靠；
4. **断网缓存**：网络未就绪时，告警数据暂存本地非易失缓冲区，待网络恢复后自动补发。

### 5. 多传感器 I²C 总线环境感知

所有板载传感器复用 **I²C1 总线**，主循环以固定时序轮询各传感器：

- **LIS2DH12** → 加速度三轴数据（碰撞判据）；
- **AHT20** → 环境温度、湿度；
- **ADC（Pin C5）** → 光照强度对应电压值。

传感器数据以 **1Hz** 频率上传至MQTT云平台，形成骑行环境日志，可供赛后运动分析或学术论文数据支撑。统一I²C总线架构既降低了引脚占用，也简化了驱动维护成本。
