## 项目概述

### 功能定位

智能骑行头盔系统旨在为骑行者提供全方位安全保障，通过多传感器融合感知骑行状态，实时监测碰撞事件，并在事故发生时自动触发 SOS 告警，将位置信息上传至云端平台，实现从"碰撞发生"到"救援信息送达"的全自动化闭环。

### 设计目标

| 目标 | 指标 |
|------|------|
| 传感器采集频率 | 加速度计 10Hz，温湿度/光照 1Hz，GNSS 1Hz |
| 碰撞检测延迟 | < 100ms |
| SOS 确认时间 | 30秒倒计时，可按键取消 |
| 数据上传周期 | 每 2 秒 |
| 通信方式 | MQTT over TLS / 4G |

## 软件架构

### 架构演进

```
V1 (单线程) → V2 (尝试子线程) → V3 (子线程栈溢出) → V4 (单线程分时轮询)
```

**架构决策说明**：

QuecPython 默认子线程栈空间为 4KB，在子线程中执行 I2C 传感器读取、AT 指令调用等操作时会触发 `maximum recursion depth exceeded` 栈溢出错误。经过多次测试验证，最终采用**单主线程分时轮询架构**，所有业务逻辑在主线程循环内执行。

### 当前架构（V4）

#### 单主线程分时轮询框架

```
┌─────────────────────────────────────────────────────────────┐
│                     主循环 (while running)                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 1. 传感器采集 (100ms 周期)                            │  │
│  │    ├── LIS2DH12 三轴加速度 → SharedData.update_acc()  │  │
│  │    ├── AHT20 温湿度 → SharedData.update_env()        │  │
│  │    └── ADC 光照强度 → SharedData.update_env()         │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 2. GNSS 更新 (5s 周期)                                │  │
│  │    └── quectel.GNSS.get_location() → 位置/速度        │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 3. 碰撞检测状态机 (50ms 周期)                          │  │
│  │    └── CollisionDetector.update() → 状态判断          │  │
│  │        └── SOS_TRIGGERED → SharedData.set_sos()       │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 4. MQTT 数据上传 (2s 周期)                            │  │
│  │    └── 温度/湿度/状态/GPS/速度 → JSON 格式发布         │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 5. SOS 告警处理 (实时)                                 │  │
│  │    ├── MQTT 紧急消息发布                               │  │
│  │    └── SMS 短信发送 (quectel.SMS)                     │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 6. LCD 显示更新 (1s 周期)                             │  │
│  │    └── 双卡片布局：环境数据 + GNSS 运动数据            │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 7. 按键中断处理                                        │  │
│  │    └── KEY 按下 → 取消 SOS 告警                        │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 线程安全机制

```
SharedData (锁保护缓冲区)
├── update_acc() / get_acc()     # 加速度数据
├── update_env() / get_env()     # 环境数据
├── set_sos() / has_sos()        # SOS 标志
└── i2c_lock                     # I2C 总线互斥锁
```

使用 `_thread.allocate_lock()` 实现中断 ISR 与主线程之间的数据同步，避免竞态条件。

### 2. 碰撞检测模块（CollisionDetector）

文件：[helmet_v4.py](helmet_v4.py)

#### 四状态有限状态机

```
IDLE ──────────────────────────────────────────────────────────┐
 │                                                              │
 │ 加速度 < 0.5g 持续 100ms                                      │
 ▼                                                              │
WEIGHTLESS ──────────────────────────────────────────────┐     │
 │                                                         │     │
 │ 加速度 > 3.5g (冲击) 或 持续失重                        │     │
 ▼                                                         │     │
POST_ACCIDENT ──────────────────────────────────────┐     │     │
 │                                                   │     │     │
 │ 30秒倒计时完成                                     │     │     │
 ▼                                                   │     │     │
SOS_TRIGGERED ──────────── 按键取消 ─────────────────┘     │     │
 │                                                        │     │
 │ 发送 MQTT + SMS 告警                                   │     │
 │                                                        │     │
 └────────────────────────── 按键取消 ─────────────────────┘     │
                                                                │
 └───────────────────────────────────────────────────────────────┘
```

**状态转换条件**：

| 状态 | 转换条件 | 下一个状态 |
|------|----------|-----------|
| IDLE | 合加速度 < 0.5g 持续 100ms | WEIGHTLESS |
| WEIGHTLESS | 合加速度 > 3.5g | POST_ACCIDENT |
| POST_ACCIDENT | 30秒倒计时结束 | SOS_TRIGGERED |
| POST_ACCIDENT | 按键按下 | IDLE |
| SOS_TRIGGERED | 按键按下 | IDLE |

## 数据格式

### MQTT 上传数据格式

```json
{
  "温度": "28.2℃",
  "湿度": "48.4%",
  "状态": "正常",
  "GPS定位": "北纬30°12'34.56\" 东经120°56'12.34\"",
  "速度": "6.2m/s"
}
```

### 状态字段说明

| 状态值 | 含义 |
|--------|------|
| `"正常"` | 无异常 |
| `"拿起"` | 头盔被拿起（加速度 > 1.2g） |
| `"自由落体"` | 检测到失重（加速度 < 0.5g） |
| `"跌倒"` | 碰撞后静止 |
| `"SOS"` | 碰撞确认，告警已触发 |

### SOS 告警消息格式

```json
{
  "type": "sos",
  "timestamp": 1688457600,
  "latitude": 30.21,
  "longitude": 120.94,
  "speed": 0.0,
  "message": "检测到碰撞，请立即救援！"
}
```

## 技术创新点

### 1. 四阶段有限状态机碰撞检测

摒弃传统单阈值判断方式，提出 "IDLE → WEIGHTLESS → POST_ACCIDENT → SOS_TRIGGERED" 四阶段状态机模型，实现对"失重-冲击-伤后静止"全过程的分阶段精确判别，有效区分正常骑行颠簸与真实碰撞事件。

### 2. "算法+交互"双重防误报机制

在多阶段状态机判定基础上，引入 30秒"二次确认"人机交互环节 —— 蜂鸣器声光提示 + LCD 倒计时 + 按键取消三重联动，解决了个体生理差异与复杂路况颠簸导致的传统设备易误报痛点。

### 3. 基于 4G MQTT 的远程 SOS 救援链路

打破传统头盔"单兵作战"的信息孤岛，在碰撞确认后自动通过 MQTT over 4G 将精确 GPS 坐标上报云端，实现从"碰撞发生"到"救援信息送达"的全自动化闭环，大幅缩短事故后急救响应时间。

### 4. 单主线程分时轮询架构

针对 QuecPython 子线程栈空间限制（默认 4KB），创新性地采用单主线程分时轮询架构，通过时间戳精确调度不同周期的任务（100ms 加速度采集、50ms 状态机轮询、2s 数据上传），避免了子线程栈溢出问题，同时保证了系统稳定性。
